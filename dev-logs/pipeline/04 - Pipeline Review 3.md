# Pipeline 三轮审计报告（顺序语义与运行时状态）

## 审计背景
- 基于此前两轮审计结论与本轮讨论，针对以下主题做聚焦复核：
  - Stage 连续性校验的顺序语义来源
  - 禁止“跳调后续 stage”的运行时约束
  - RuntimeState 全局注册方案的隔离性风险
- 本次仅形成审计结论与规范建议，**不包含代码修改**。

## 当前基线
- 相关实现文件：`src/alpenstock/pipeline/_decorators.py`、`src/alpenstock/pipeline/_meta.py`
- 相关文档：`01 - Pipeline Draft.md`、`docs/guides/pipeline.md`
- 相关测试：`tests/pipeline/*`、`tests/typecheck/*`
- 测试状态：现有用例通过（此前基线 `pixi run pytest -q` 为通过状态）

## 发现（按严重度排序）

### 1. [高] 连续性校验依赖“推断顺序”，缺少稳定且显式的 Stage 顺序定义

**现象**
- 框架在做 stage 快照连续性校验时，需要一个可比较的“先后顺序”。
- 现状顺序来源受 MRO 与方法可见性影响，不等价于用户的实际执行语义。

**风险**
- 继承/覆盖/重排场景下，合法缓存可能被误判为损坏。
- 文档与实现难以给出一致、可预期的恢复语义。

**结论**
- 若保留连续性校验，框架必须有唯一且稳定的 Stage 顺序来源，不能继续依赖隐式推断。

---

### 2. [高] 允许直接调用后续 Stage，会产生“先写坏布局、后续启动报错”的行为

**现象**
- 用户可在冷启动时直接调用后续 stage。
- 该行为可能写入仅包含后续 stage 的快照，下一次 bootstrap 时触发连续性错误。

**风险**
- 用户在第一次调用时看似成功，后续才报“缓存损坏”，可理解性差。
- 形成“延迟失败”语义，调试成本高。

**结论**
- 需要运行时顺序约束：Stage 必须按定义顺序执行，禁止跳调。

---

### 3. [中] RuntimeState 使用全局表（按 `id(instance)`）存在跨实例串扰窗口

**现象**
- 运行时状态通过全局字典维护，键为 `id(instance)`。
- 在实例不可 weakref 的情况下，旧 runtime 条目可能无法及时清理。
- 当 Python 复用对象 id 时，新实例可能误取旧 runtime。

**风险**
- 新实例可能继承旧实例的 `cache_enabled/save_path/finished_stages` 等内部状态。
- 造成错误缓存命中、错误跳过 stage、或路径污染。

**结论**
- RuntimeState 更适合实例内持有（懒初始化属性），不应依赖全局 id 映射。

## 决议建议（用于下一步实现）
1. `stage_func` 增加显式参数 `order: int`，数值越小越早执行。
2. `order` 作为唯一排序语义来源（`stage_id` 保留为标识与文件名用途）。
3. 同一 Pipeline 的 `order` 必须唯一；冲突时类构建阶段报错。
4. 运行时增加顺序守卫：只允许调用“下一阶段”。
5. RuntimeState 改为实例级存储（懒初始化），移除全局 id 注册表方案。

## 影响评估（初步）
- API 变更：`@stage_func(id="...")` 将升级为 `@stage_func(id="...", order=...)`。
- 文档变更：草案与用户指南需补充顺序规则、跳调错误语义、继承行为约定。
- 测试变更：
  - 新增 `order` 唯一性/类型校验用例
  - 新增“跳调后续 stage 立即报错”用例
  - 新增实例级 runtime 隔离回归用例

## 结论
- 本轮讨论已把“顺序语义”和“运行时状态隔离”两个根问题明确化。
- 推荐先完成草案修订，再按修订后的契约实施代码与测试改造。
